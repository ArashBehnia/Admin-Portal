1. Purpose and scope
This document is the frontend engineering specification for the HomeBy Admin Portal-Phase 1. It tells you what to build, how to wire it, and what APIs to call. It does not explain why decisions were made or teach general React patterns. Read the Product Overview document first if you need the business context.
Phase 1 covers seven sections:
Dashboard
Integrations & Feed Operations
Agencies (list + detail)
Agents (directory)
Applications (moderation queue)
Email Templates
Staff & Roles
The Lovable prototype at the URL provided by Arash is the visual reference for every page. Pixel-perfect match is not required-functional accuracy and design system compliance are. When in doubt, follow the prototype.

3. Authentication
3.1 Flow
Login is two steps: password → TOTP MFA. Both steps hit the admin backend. No third-party auth UI.
Step
API call
1. Submit email + password
POST /auth/login → returns { mfa-required: true, session-token }
2. Submit 6-digit TOTP code
POST /auth/mfa/verify → returns { access-token, refresh-token, role, expires-at }
3. Forgot password
POST /auth/password/reset/request → sends email with reset link
4. Token refresh
POST /auth/token/refresh → call before expiry using refresh-token
5. Logout
POST /auth/logout → invalidates server-side session
3.2 Token storage
Store access-token and refresh-token in memory (React context). Do not use localStorage or sessionStorage in this app-security requirement. On page refresh, redirect to /login.
3.3 Auth context
interface AuthContext {
  role: 'superadmin' | 'admin' | 'support' | 'reviewer' | 'content-editor';
  staff-id: string;
  name: string;
  access-token: string;
  logout: () => void;
}
3.4 Role-based UI
The sidebar hides items based on role. Only superadmin sees Audit Log. Implement with a simple role check in the sidebar component. Other sections enforce access at API level-the frontend just hides the nav items.
3.5 Axios interceptor
All requests must include the Authorization header. On 401 response, attempt token refresh once, then redirect to /login.
axios.defaults.baseURL = import.meta.env.VITE-ADMIN-API-URL;
// Request interceptor: attach Bearer token from auth context
// Response interceptor: on 401 → refresh token → retry → else logout
4. Shared component patterns
Every list view, detail page, and moderation queue follows the same layout. Build these as reusable components once-use them everywhere.
4.1 List page pattern
Summary stat tiles row (KPI cards)-3 or 4 cards, consistent height
Filter row: search input left + filter pills right
Data table: sortable columns, 44px rows, status badges, action links
Pagination: page controls bottom-right, rows-per-page selector
Empty state: centered icon + heading + subtext + optional CTA
4.2 Detail page pattern
Header card: entity name, key badges, inline stats (4 items), action menu (kebab)
Horizontal tab bar below header
Tab content area: full width, consistent padding
Audit tab: always the last tab on every detail page
4.3 Side panel pattern
400px wide, slides in from right, overlays main content (no navigation away)
Panel header: name, badges, close X button
Panel tabs if needed
Action buttons at bottom, stacked, full width
Destructive actions: require typed confirmation modal before executing
4.4 Moderation queue pattern
Tabs across top: Pending (default, with badge) / Approved / Rejected / All
Table below: filter + search row, results, side panel on row click
Nothing publishes or changes status without an explicit admin action
4.5 Status badges
Standardise these across the entire app:

Rendering rules

6. Error handling
Scenario
Behaviour
401 Unauthorized
Attempt token refresh once. If refresh fails, redirect to /login with session expired message.
403 Forbidden
Show inline error: 'You do not have permission to perform this action.'
404 Not Found
Show empty state with 'Record not found' message. No crash.
500 Server Error
Show toast: 'Something went wrong. Try again or contact support.' Log to console.
Network timeout
Show toast with retry button. Do not auto-retry silently.
Form validation
Inline errors below each field. Red border on invalid field. Error text in danger colour.
